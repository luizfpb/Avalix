# BodyTrack — V1.1 (testar, configurar e melhorar)

Cole este arquivo junto com `docs/DECISIONS.md` no início de uma conversa nova. O DECISIONS diz **o que o programa é**; este aqui diz **o que falta fazer** antes do beta e as ideias de melhoria. Em Claude Code, peça pra ler `docs/DECISIONS.md`, `docs/V1.1.md` e `src/`.

## Onde paramos
O programa está **funcionalmente completo pra V1/beta**: cadastro de avaliados, consentimento eletrônico, avaliação física (motor de protocolos testado + formulário com cálculo ao vivo), postural (captura/upload com compressão + comparação), relatórios (PDF + CSV + auditoria), 2FA (MFA TOTP) e operação (workflows de keep-alive e backup).

Estado técnico nesta conversa: `npm run lint`, `npm run build` e `npm run test` (67 testes) **passando**. Tudo commitado e no GitHub.

O que **ainda não foi feito**: ninguém testou o app em runtime de ponta a ponta (auth já foi na 3.2, o resto não), os secrets dos workflows não estão configurados, e há decisões/correções listadas abaixo. A v1.1 é exatamente isto: **testar o essencial, configurar, e aplicar as melhorias que valerem a pena**.

## Parte 1 — Testar (só o essencial, em ordem)

Como testar, pra quem não faz isso direto: rode `npm run dev`, abra `http://localhost:5173`. Se algo der errado, abra o **DevTools** (tecla F12), aba **Console** (mensagens de erro em vermelho) e aba **Network** (requisições que falharam aparecem em vermelho/status 4xx-5xx). Copie a mensagem e traga pra conversa. Não precisa decorar nada; o erro quase sempre diz o que faltou.

Teste nesta ordem (cada passo depende do anterior):

1. **Login** — já validado na etapa 3.2; só confirme que entra e cai no dashboard.
2. **Cadastrar avaliado** — Avaliados > Novo. Cadastre um adulto e um menor de 18 (o menor deve exigir responsável). Certo: aparece na lista; o menu mostra o termo que você escolheu na org (ex.: "Alunos").
3. **Consentimento** — no detalhe do avaliado, "Ler o termo" (confira que o nome da sua org aparece no texto), registre. Certo: vira "vigente"; teste "Revogar".
4. **Avaliação física** — com consentimento vigente: Nova avaliação > JP7 > peso/altura + 7 dobras. Certo: o resultado calcula ao vivo (% gordura, massas). Salve, veja no histórico, abra o detalhe.
   - **Baixar PDF**: o primeiro clique baixa um arquivo grande (~1,5 MB, é a biblioteca de PDF) e depois gera. Confira se os **acentos** saem certos no PDF.
   - **Exportar CSV / Excel BR**: abra o "Excel BR" no Excel e veja se as colunas vêm separadas.
5. **Postural (maior risco — primeira vez que o Storage é usado)** — Nova sessão > Adicionar foto. Certo: a foto sobe, aparece como thumbnail, e clicar abre ela cheia. Depois teste **Comparar** (precisa de ≥2 fotos): lado a lado e sobreposição com o slider.
   - Se o upload falhar: ver Parte 2.1 (o bucket `photos` precisa existir) e confirme que há consentimento vigente.
   - **Teste no celular real** (de preferência um iPhone), porque a compressão usa uma API de imagem que se comporta diferente no Safari.
6. **MFA (2FA)** — Configurações > Ativar 2FA, escaneie o QR no Google Authenticator/Authy, confirme o código. Depois **deslogue e logue de novo**: tem que cair na tela pedindo o código. Certo: só entra no app depois do código. Se entrar em loop ou não pedir, traga o caso.

Não precisa testar mais que isso. O resto (edição, exclusão de foto, filtros) é secundário.

## Parte 2 — Configurar (explicado passo a passo)

### 2.1 Habilitar o MFA no Supabase
Painel do Supabase > **Authentication** > **Multi-Factor Auth** (ou "MFA") > garanta que **TOTP / Authenticator app** está habilitado. Sem isso, "Ativar 2FA" no app dá erro.

### 2.2 Secrets dos workflows (GitHub)
Os dois workflows (`.github/workflows/keep-alive.yml` e `backup.yml`) já estão no repositório e **vão rodar nos horários agendados — mas falham sem os secrets**. Configure em: repositório no GitHub > **Settings** > **Secrets and variables** > **Actions** > **New repository secret**. Crie cada um:

- `SUPABASE_URL` → painel Supabase, Settings > API > **Project URL**.
- `SUPABASE_PUBLISHABLE_KEY` → Settings > API Keys > a chave **publishable** (`sb_publishable_...`).
- `SUPABASE_DB_URL` → Settings > **Database** > Connection string > **URI**. Use a opção **Session pooler** (não a "Direct connection"): o runner do GitHub é IPv4 e a conexão direta costuma ser IPv6. Copie a string inteira (já vem com a senha; se mostrar `[YOUR-PASSWORD]`, troque pela senha do banco que você salvou).
- `BACKUP_PASSPHRASE` → invente uma senha forte só pra criptografar o backup e **guarde no gerenciador de senhas** (sem ela o backup é inútil).

### 2.3 Testar os workflows agora (sem esperar o agendamento)
No GitHub, aba **Actions** > escolha o workflow (ex.: "keep-alive") > botão **Run workflow**. Veja se passa verde. Faça o mesmo com "backup" — ele deve gerar um **artifact** chamado `db-backup` (aparece no resumo da execução, baixável).

### 2.4 Teste de restore (o passo mais técnico — pode fazer junto comigo numa conversa)
Backup que nunca foi restaurado não é backup. A ideia: baixar o artifact, descriptografar e restaurar num **projeto Supabase descartável**. Comandos (precisa do `psql`/`gpg` instalados, ou peça ajuda numa conversa):
```
gpg --batch --passphrase 'SUA_BACKUP_PASSPHRASE' --decrypt backup-XXXX.sql.gpg > dump.sql
psql "CONNECTION_STRING_DO_PROJETO_DESCARTAVEL" < dump.sql
```
Se preferir, deixe esse passo pra fazermos juntos — é o mais chato e o que mais erra.

## Parte 3 — Revisão do código (o que eu olhei com cuidado)

Prioridade: **P0** = decidir/corrigir antes do beta; **P1** = importante; **P2** = bom ter.

- **[P0] [FEITO v1.3] O MFA é só uma barreira no cliente, não no banco.** Resolvido na `supabase/migrations/0003_mfa_aal2.sql`: `app.mfa_satisfied()` exige AAL2 no JWT, mas só quando a conta tem fator MFA verificado (conta sem 2FA segue funcionando). Checado em `can_view_subject` (cobre leitura de todo dado sensível + Storage) e nos `with check` de subjects. **Falta aplicar a migration no Supabase e testar o 2FA ponta a ponta.**
- **[P0] Texto de consentimento é rascunho (`0.1-rascunho`), sem revisão jurídica.** Não coletar dado de titular real com ele antes de revisar. Quando aprovar, suba a versão pra `1.0` (o hash acompanha).
- **[P1] [FEITO v1.3] Exclusão definitiva de avaliado (direito de eliminação da LGPD).** `deleteSubjectCompletely` remove os arquivos do Storage **antes** das linhas; o cascade leva o resto e a auditoria registra o DELETE. UI: zona de perigo no editar, liberada só ao digitar o nome exato.
- **[P1] Captura de foto no Safari/iOS.** A compressão usa `createImageBitmap` com correção de orientação; funciona em navegadores modernos, mas precisa ser testada no aparelho real da avaliadora. Se quebrar lá, dá pra fazer um fallback. **Testar (Parte 1.5).**
- **[P2] [FEITO v1.2] Sem "error boundary".** Adicionado `src/components/ErrorBoundary.tsx`, instalado no `main.tsx`: tela amigável com botão de recarregar no lugar do branco.
- **[P2] [FEITO v1.4] Bundle inicial ~700 KB.** Todas as rotas agora entram via `React.lazy` (App.tsx) + Recharts e PDF já eram lazy. Bundle inicial caiu de 706 KB para 489 KB.
- **[P2] [FEITO v1.3] Exclusão de sessão postural inteira** (`deleteSession` + botão "Excluir sessão" no detalhe da sessão), que também resolve as **sessões vazias** abandonadas.

Já corrigido nesta conversa: lint zerado (config + `_`-prefixados), e as **URLs assinadas das fotos agora renovam a cada 4 min** (antes a imagem quebraria depois de 5 min de tela aberta).
